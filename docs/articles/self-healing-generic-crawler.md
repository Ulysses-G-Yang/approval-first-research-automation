# 我手搓了一个会“自我修复”的通用爬虫：三层容灾机制是如何炼成的

在做网页数据采集时，最麻烦的往往不是第一次跑通，而是页面小改版后整条链路突然失效。这个项目最初是一个单站点脚本，后来重构为配置驱动的通用框架：把 URL、分页、字段和页面动作放进 YAML，把浏览器控制、提取和异常处理留在稳定的执行引擎里。

## 问题分析

硬编码脚本容易把页面结构、字段语义和业务规则绑在一起。一个 selector 改动，就可能让整个任务失去输出；排查时又难以判断是页面变化、加载失败还是数据格式变化。

框架的目标不是承诺任何页面都能自动恢复，而是把失败变成可以观察、分层处理和回归验证的工程问题。

## 三层容灾

第一层是配置中的 CSS selector，路径清晰、速度快，也是最容易维护的一层。

第二层是 Scrapling 自适应解析。当常规 selector 没有结果时，引擎会基于页面 HTML 尝试自适应定位。它适合吸收轻微结构变化，但不会替代针对真实页面的验证。

第三层是可选的 LLM 选择器修复。只有在前两层均失败且配置明确开启时，系统才会把字段语义和受限 HTML 上下文交给模型，获得候选 CSS selector 后仅在当前页面重试一次。结果会被缓存，失败则记录日志并返回空值，不阻塞任务。

## 工程边界

LLM 修复默认关闭，避免意外费用和不可预期输出。验证脚本通过正常配置和故障注入检查关键字段，而不是只统计记录数量。离线单元测试覆盖解析、缓存、超时和降级；真实站点验证仍应在有权限、遵守条款的场景下手动进行。
